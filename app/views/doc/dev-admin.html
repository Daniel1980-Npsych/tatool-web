Administration

It's important to familiarise yourself with the functionality offered by the Administrator role, as this will help you to manage Users and Projects of your local Tatool instance whenever you need to.

The first time Tatool is started, a default Admin user will be added to allow you to login and setup your own users (see Getting Started for more details). You can safely delete the default Admin user as long as you provide another user with the Admin role. If you end up deleting all Admin users you can simply restart the server and the default Admin user will be added again.

User Management

In order to use Tatool, you'll need to have a user and password assigned to you. When running Tatool in lab mode there is no user registration feature. The only person able to add/edit/remove users is the Administrator. When logged in as an Administrator, you'll have access to the Admin tab in the top navigation which will lead you to the User Management screen.

On the User Management screen you can add, edit or delete users. The most important part is the role assignment for every user. Tatool supports the following Roles:

  • Modules: Access to the Modules tab which allows installing and running Modules. Every user should have this role assigned.
  • Editor: Access to the Editor tab which allows to create and publish Modules.
  • Analytics: Access to the Analytics tab which allows to monitor usage of published Modules.
  • Admin: Access to the Admin tab which allows to manage Users and Projects.

Project Management

A Project groups all the files needed for a specific experiment into one folder. For example, every Executable is part of a Project, and every stimuli file or stimulus you use is part of a Project. There is one special type of Project which is called External Resource which does what it says: it represents a resource from an external source (e.g., some http address) instead of a local Tatool Project.

An Administrator can add/edit/remove Projects. A Project is defined by the following attributes whereas the Name, Access and Owner property can't be changed after the Project is created:

  • Name: A unique name which will represent this Project (= folder name)
  • Access: A Project can either be Public which means every developer can have access to its resources, or it can be Private which means only the owner of the Project can use its resources.
  • Owner: The owner of a Project represents the login email of the user owning a Project.
  • Description: Allows to provide a small description of this Project. This will be displayed in the Module Editor whenever you select a Project.
  • Executables (JSON): If you're planning on having your own Executables as part of a Project you can provide their details as a JSON String. This allows you to select the Executables from a dropdown list in the Module Editor. The JSON string is an array of objects whereby every object has the property customType identifying the Executable Service name and a small description which will be shown in the Module Editor:
    [ { "customType" : "myExecutable", "description" : "A small description of the Executable" }, 
{ "customType" : "myOtherExecutable", "description" : "A small description for this Executable" } ]

In order to physically add resources (e.g. Executables, stimuli files) to a specific Project, you'll have to open the tatool-web folder on the server running your Tatool instance. Within that folder you'll find the corresponding Project folders under /app/projects/*. Every Project folder contains 4 mandatory sub-folders that have to be used to save specific file types:

  • executables: Tatool will search for your custom Executables within this folder.
  • instructions: The Tatool Instruction Executable will search for instruction html files within this folder.
  • modules: A folder for you to keep any Module files belonging to a specific Project.
  • stimuli: A folder to keep any stimuli files provided by a Project.
